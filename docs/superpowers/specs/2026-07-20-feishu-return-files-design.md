# 飞书文件回传设计

## 目标

让 Codex 在飞书会话中生成 HTML、PDF、Office 文档或其他本地文件后，能够把文件上传到飞书并回复到触发任务的原始消息。

## 当前缺口

`src/feishu/send.ts` 已经实现飞书文件上传、回复和主动发送，但 `FeishuChannel` 收到 Codex 输出后只发送文字，没有解析文件路径，也没有调用 `sendFeishuFile`。

## 输出协议

Codex 使用单独一行指令声明待回传文件：

```text
[[codex:file:相对或绝对路径]]
```

飞书输入提示会附带这条协议说明。为兼容 Codex 自然回复，网关也识别回复中当前工作目录下已经存在的绝对文件路径。协议指令和识别出的路径会从用户可见文字中移除。

## 数据流

1. 飞书消息进入 `FeishuChannel`，记录当前消息 ID。
2. `CodexSessionRouter.send` 接收本次消息专属的输出回调，并在队列轮到该请求时执行 Codex。
3. Codex 返回文字后，router 调用该请求自己的输出回调，避免并发消息串到错误的回复目标。
4. channel 解析 `[[codex:file:路径]]` 和工作目录内的绝对文件路径。
5. channel 先回复清理后的文字，再逐个调用已有 `sendFeishuFile` 上传并回复文件。

## 手动命令

支持以下命令，不经过 Codex：

```text
/file 路径
/sendfile 路径
```

路径相对于当前飞书账号的 Codex 工作目录解析。

## 安全边界

- 只能回传当前账号工作目录内的文件。
- 文件必须存在、是普通文件且非空。
- 单个文件不能超过 30MB。
- 文件路径去重，避免同一个文件重复上传。
- 不读取或回传工作目录外的路径。

## 错误处理

- 无效自动路径被忽略并写入服务日志，其他有效文件仍继续发送。
- 飞书上传失败沿用现有消息处理错误链路，向原消息回复处理失败原因。
- 只有文件没有文字时，不额外发送空白文字消息。

## 测试

- 路径解析：协议指令、绝对路径、代码块清理和去重。
- 安全校验：目录越界、不存在、空文件、目录和超过 30MB。
- channel：Codex 输出自动回传、手动 `/file`、仅文件输出和原消息绑定。
- router：同一会话排队请求各自使用对应的输出回调。
- 回归：完整测试、类型检查和构建。
